# Skills 目录产品指南

> 用于规划、实现和维护当前 Skills 展示站点。

---

## 1. 产品意图

这个站点是一个 Skills 的“发现与选择层”，不是“教学层”。

它的职责只有三件事：

- 帮用户理解这是什么
- 帮用户判断是否适合自己
- 帮用户跳转到原始来源继续了解

如果某个内容区域不能帮助用户完成这三件事，就应该质疑它是否有必要存在。

---

## 2. 产品边界

### 2.1 应该做的

- 展示多个 Skill
- 支持用户快速搜索与筛选
- 卡片化呈现关键信息
- 跳转 GitHub
- 支持维护者手动更新数据
- 支持通过 GitHub URL 半自动生成卡片数据
- 支持中英文双语显示

### 2.2 不应该做的

- 不写安装教程
- 不写完整使用文档
- 不做用户账号系统
- 不做访客提交或编辑
- 当前版本不做后台 CMS
- 当前版本不做复杂推荐算法

---

## 3. 页面结构规则

页面结构应保持浅层、清晰、可快速扫描。

推荐结构：

1. Hero 介绍区
2. 搜索与筛选区
3. 卡片网格区

当前已确认：

- 不使用左侧筛选栏
- 筛选统一放在卡片上方
- 公开页面不展示维护说明或数据说明模块
- 页面整体保持简洁，不再频繁改版

---

## 4. 卡片设计规则

每张卡片都应快速回答以下问题：

1. 这是什么
2. 它主要做什么
3. 适合谁
4. 来源是什么
5. 我要点哪里

### 4.1 推荐字段

- `name`
- `description`
- `audience`
- `source`
- `category`
- `tags`
- `github_url`

### 4.1.1 名称与来源边界

- `name` 只展示 Skill 本身的名称，不重复展示作者、组织或仓库路径
- 如果原始名称是 `owner/repo/skill`、`owner/skill` 或类似路径形式，公开卡片名称只保留最后一段
- 作者、组织、仓库和目录路径统一放在 `source`
- 示例：`NVIDIA/video-search-and-summarization/report` 的 `name` 应为 `report`，`source` 保留完整来源路径

### 4.2 文案规则

- 只保留一句简介
- 不再保留独立 `use_case`
- 简介要直接说明作用
- 不写操作步骤
- 不写安装引导
- 不写过度宣传式语句

错误方向：

- “一键安装……”
- “快速上手……”
- “最强、最佳、终极……”

正确方向：

- “用于……”
- “帮助用户……”
- “适合在……场景下使用”

---

## 5. 双语数据规则

为避免中英文仓库混杂显示，站点应逐步演进为“双语字段 + 前端切换”。

当前阶段先实现界面文案双语切换：

- 页面固定文案支持 `中文 / English` 两个按钮切换
- 卡片字段名、按钮、空状态、反馈区文案随语言切换
- 前端兼容字符串字段和 `{ zh, en }` 双语对象字段
- 现有字符串数据按原文显示，缺失当前语言时回退到 `zh`、`en` 或原字符串
- 新生成卡片的 `name`、`description` 应使用 `{ zh, en }` 结构
- 当前不会在前端自动翻译卡片内容，英文字段需要生成或人工校正

推荐结构：

```js
{
  name: {
    zh: "中文名称",
    en: "English Name"
  },
  description: {
    zh: "中文简介",
    en: "English description"
  },
  audience: "开发",
  source: "owner/repo",
  category: "工作流",
  tags: ["session", "viewer"],
  github_url: "https://github.com/owner/repo"
}
```

### 4.3 来源字段规则

- 公开卡片不再展示 `level` / 难度
- `source` 用来标注 GitHub 来源作者、组织或仓库路径
- 普通仓库使用 `owner/repo`
- 仓库内具体 Skill 目录使用 `owner/repo/path`
- `source` 只用于展示和追溯来源，当前不作为筛选维度

规则：

- 页面默认显示中文
- 允许用户切换为英文
- `name` 与 `description` 使用双语对象
- 结构化字段继续使用固定值或数组
- 维护者可以手动修正文案

## 6. 分类与筛选规则

为了帮助用户精准、快速锁定 Skill，当前采用“搜索 + 两个辅助筛选维度”的模型。

### 6.1 第一层：类别

这是主要筛选维度，用于说明“它是做什么的”。

建议控制在少量类别：

- 文档
- 开发
- 工作流
- 插件
- 图像
- 自动化
- 研究
- 嵌入式

### 6.2 第二层：适用人群

这是辅助维度，当前只保留三类：

- 开发
- 设计
- 工作流

### 6.3 筛选顺序

推荐顺序：

1. 搜索框
2. 类别
3. 适用人群

### 6.4 高频来源标签

当同一个 GitHub 组织或作者收录的 Skill 数量超过 15 个时，在筛选区下方显示来源标签。也可以固定展示少数重点来源。

规则：

- 来源标签内容使用 `source` 字段的第一段，即 `owner/repo` 中的 `owner`
- `gitee:` 前缀只作为来源协议标记，计算标签时不展示
- 标签只作为快捷筛选入口，不替代类别和适用人群筛选
- 点击来源标签后，只显示该作者或组织下的 Skills
- 保留“全部来源”标签用于清除来源筛选
- 当前固定展示来源：`anthropics`、`zuoliangyu`

## 7. 数据维护规则

当前版本的数据维护方式非常明确：

- 只有维护者可以更新数据
- 观众只能浏览
- 数据源集中在 `data.js`
- 当前不引入后端

维护流程：

```text
维护者修改 data.js
      |
      v
页面自动读取展示
      |
      v
观众只读访问
```

---

## 8. 半自动卡片生成规则

如果维护者提供一个 GitHub URL，系统应尽量直接生成卡片数据并写入 `data.js`，而不是要求维护者手动搬运中间文件。

自动生成不等于放弃人工控制。写入后，维护者仍然可以继续手动修改卡片字段。

推荐流程：

```text
GitHub URL
  ->
抓取公开信息
  ->
优先读取 README / SKILL / description / topics
  ->
提炼一条标准简介
  ->
生成卡片数据
  ->
直接写入或更新 data.js
  ->
维护者按需要继续手动修改
```

规则：

- 自动生成的目标是减少维护者手填成本
- 自动写入后仍允许维护者继续手动修正
- 对同一个 `github_url`，应更新原卡片，不应重复追加
- 观众没有新增、编辑、删除卡片的权限
- 页面层不承载抓取逻辑

---

## 9. 自动发现候选规则

自动发现候选 Skill 时，不应全 GitHub 无差别抓取，也不应把候选直接写入 `data.js`。

推荐方式：

```text
GitHub Search
  ->
候选仓库
  ->
规则打分
  ->
candidates/skills_candidates.json
  ->
维护者确认
  ->
正式生成卡片
```

候选仓库应按信号打分：

- `SKILL.md` 是强信号
- topic 包含 `skill`、`claude-code`、`codex`、`ai-agent` 是强信号
- README 提到 Claude Code、Codex、Agent Skill 是中信号
- 存在 `.codex`、`.claude`、`skills`、`agents` 目录是中信号
- 仅仅是普通 AI 应用、普通 SDK、普通库，不应直接视为 Skill

判断标准：

```text
它是否为某个 Agent / AI 工作流提供可复用能力？
```

只有候选确认后，才进入正式卡片生成流程。

---

## 10. 质量判断标准

一个合格的 Skills 卡片，应满足：

- 用户 3 秒内能看懂它是做什么的
- 用户不用读长文就能判断是否值得点开
- 用户能明确看到类别、适用人群和来源
- 用户能快速与其他卡片做横向比较

一个合格的 Skills 页面，应满足：

- 用户进入首页后能立即知道这是一个 Skills 展示站
- 用户能通过少量筛选快速缩小范围
- 页面看起来干净，不像文档站，也不像后台管理页

---

## 11. 实现优先级

当前推荐优先级如下：

1. 固定页面结构
2. 固定字段模型
3. 固定筛选模型
4. 固定手动维护流程
5. 做好 GitHub URL 卡片生成
6. 升级为双语数据结构
7. 增加前端语言切换
8. 优化自动生成质量与重复链接处理
